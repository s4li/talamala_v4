# بسته بررسی عملیاتی — گلدیس هاب v2.7

## هدف این بسته

این مجموعه مستندات برای **بررسی عملیاتی روال های سیستم گلدیس هاب** قبل از شروع کدنویسی تهیه شده. هدف اینه که مدیران واحدهای مختلف شرکت، هر روال رو از دید عملیاتی و بیزینسی بررسی کنن و نظراتشون رو قبل از پیاده سازی اعلام کنن.

**این مستندات فنی نیستن.** هیچ کد، SQL یا جزئیات API توشون نیست. زبان ساده و قابل فهم برای مدیران غیرفنی استفاده شده.

## جلسات بررسی چطور برگزار میشن؟

### قبل از جلسه
1. **review-sheets** مربوط به جلسه برای شرکت کنندگان ارسال بشه (حداقل ۲ روز قبل)
2. هر مدیر بخش خودش رو بخونه و سوالاتش رو آماده کنه

### در جلسه
1. هر روال حداکثر **۱۵ دقیقه** وقت داره (۵ دقیقه ارائه + ۱۰ دقیقه بحث)
2. **سوالات بخش ۹** هر review sheet مبنای بحث هستن
3. هر تصمیم، مشکل یا درخواست تغییر **همون لحظه** ثبت بشه

### بعد از جلسه
1. صورتجلسه با قالب `MEETING_MINUTES_TEMPLATE.md` تکمیل بشه
2. مسائل باز در `OPEN_ISSUES_LOG.md` ثبت بشن
3. درخواست تغییرات در `CHANGE_REQUESTS_LOG.md` ثبت بشن
4. وضعیت هر روال در `FLOW_REVIEW_STATUS.md` بروزرسانی بشه

## ثبت بازخورد

مدیران میتونن بازخوردشون رو به ۳ شکل ثبت کنن:

| نوع بازخورد | کجا ثبت بشه | مثال |
|-------------|-------------|------|
| **مشکل یا ابهام** | `OPEN_ISSUES_LOG.md` | «مشخص نیست چه کسی مسئول تایید بازخرید حضوری هست» |
| **درخواست تغییر** | `CHANGE_REQUESTS_LOG.md` | «سقف برداشت روزانه باید قابل تنظیم باشه» |
| **تایید روال** | `FLOW_REVIEW_STATUS.md` | وضعیت → Approved for Implementation |

## بعد از بررسی چه اتفاقی میافته؟

```
بررسی عملیاتی (شما اینجایید)
    ↓
اعمال تغییرات تایید شده در مستندات فنی
    ↓
بررسی نهایی توسط مدیریت ارشد
    ↓
شروع کدنویسی و پیاده سازی
    ↓
تست عملیاتی (UAT) با همکاری واحدها
    ↓
استقرار تدریجی
```

هر تغییری که مدیران درخواست کنن و تایید بشه، **قبل از شروع کدنویسی** در مستندات فنی اصلی (`flows/`، `03-schema-index.md`، `04-api-index.md`) اعمال میشه. هیچ روالی بدون تایید واحد مربوطه وارد مرحله پیاده سازی نمیشه.

## سوالات کلیدی که هر مدیر باید جواب بده

هنگام بررسی هر روال، این سوالات رو در ذهن داشته باشید:

1. **آیا این روال با واقعیت عملیات شرکت هماهنگ است؟**
2. **مسئول هر مرحله کیست؟**
3. **چه تاییدهایی لازم است؟** (دستی یا اتومات)
4. **چه مدارکی باید ثبت بشه؟**
5. **حالت های خطا چیست و چطور مدیریت بشه؟**
6. **کجا ریسک مالی، انباری یا سوءاستفاده وجود داره؟**
7. **چه اصلاحاتی قبل از کدنویسی لازمه؟**
